# ADR-011: Use template terminology consistently

| Status   | Date       | Implementation Plan                                           |
| -------- | ---------- | ------------------------------------------------------------- |
| Accepted | 2026-04-16 | [ADR-011_IMPLEMENTATION-PLAN.md](./ADR-011_IMPLEMENTATION-PLAN.md) |

## Context

The project currently uses both `preset` and `template` for the same scaffold-selection concept.

The runtime template layout already centers on `templates/base/template.ts` and `templates/base/files`, but some source modules, types, prompt answers, tests, and active documentation still use `preset` terminology. This makes the domain language harder to follow and creates unnecessary friction before adding more scaffold variants.

Historical ADRs and changelog entries may contain `preset` because they describe prior decisions and release history.

## Decision

Use `template` as the active project term for scaffold variants.

Runtime code, internal types, tests, user-facing CLI text, and active documentation will use `template` instead of `preset`. Internal names will be renamed cleanly without deprecated `preset*` aliases because the package exposes a CLI, not a public TypeScript library API.

Historical ADRs, ADR implementation plans, and changelog entries will keep their original terminology unless a link or code reference must change.

## Consequences

Positive:

- Source names, generated-asset layout, and documentation use one term.
- Future template additions have a clearer extension path.
- Developers do not need to distinguish between `preset` metadata and `template` definitions.

Negative:

- The rename touches multiple modules, tests, and active docs.
- Any hidden internal imports of `preset*` names will break.
- `rg preset` will still find historical records by design.

## Alternatives Considered

- Keep `preset` in code and update only documentation: rejected because the mixed domain model would remain.
- Add temporary `preset*` aliases: rejected because aliases preserve the terminology split without a known compatibility need.
- Rewrite historical ADRs and changelog entries: rejected because those records should preserve the wording used by past decisions and releases.

## Reversibility

Rollback by restoring the previous `preset*` module, type, field, test, and documentation names from the implementation diff.

If a later decision needs both concepts, supersede this ADR and introduce explicit definitions for how templates and presets differ.
