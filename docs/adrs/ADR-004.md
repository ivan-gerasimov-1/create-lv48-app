# ADR-004: Use tsdown for CLI builds

| Status   | Date       |
| -------- | ---------- |
| Accepted | 2026-04-15 |

## Context

The project currently builds production TypeScript with:

```bash
tsc -p tsconfig.build.json
```

This emits JavaScript and declarations into `dist/` using the source tree layout. The published CLI wrapper in `bin/create-lv48-app.js` imports `../dist/cli.js`, so any build system change must keep that runtime entrypoint stable.

The current build setup is simple and type-safe, but it does not provide a fast bundler-oriented watch mode for CLI development. The project needs a development mode that rebuilds the CLI quickly while still reporting TypeScript type errors, and a production build path that cannot silently skip type checking.

## Decision

Use tsdown for CLI build output.

Build one ESM CLI bundle from `src/cli.ts` into `dist/cli.js`, and emit bundled declarations into `dist/cli.d.ts`. Keep the existing package type and bin wrapper contract unchanged.

Keep TypeScript type checking as a separate explicit step:

- `build:typecheck` runs `tsc -p tsconfig.build.json`
- `prebuild` runs `npm run build:typecheck`
- `build` runs `tsdown`

Use `concurrently` for development mode. `dev` runs the tsdown build in watch mode by passing `--watch` through `npm run build`, and runs TypeScript in watch mode through `npm run build:typecheck -- --watch --preserveWatchOutput`.

Add `start` as a local CLI command that runs the built CLI through the existing bin wrapper:

```bash
node bin/create-lv48-app.js
```

Do not add `exports`, `main`, or `types` as part of this migration. The package remains CLI-only.

Do not use `vite-plugin-checker` for this project. The chosen development contract is terminal diagnostics from `tsc --watch` next to tsdown watch output.

## Consequences

Positive:

- Production output has fewer generated files.
- CLI development gets a fast rebuild loop.
- Type checking remains explicit and visible in both production build and development watch mode.
- The published bin wrapper continues to import `dist/cli.js`.

Negative:

- Development mode runs two long-lived processes.
- Single-bundle stack traces are less directly mapped to individual source modules when sourcemaps are disabled.
- `prebuild` must stay in place so production builds do not become transpile-only.

## Alternatives Considered

- Keep `tsc` as the only build tool: lowest dependency count and current behavior, but it does not provide the desired bundler watch workflow.
- Use tsdown with `unbundle`: closer to the current `dist/` layout, but keeps more generated files and gives less benefit from switching to a bundler.
- Use `vite-plugin-checker`: useful in Vite apps, but this package is a Node CLI and does not need Vite dev server overlay behavior.
- Add a separate `build:watch` script: explicit, but duplicates the existing `build` command surface. Passing `--watch` through `npm run build` keeps the script set smaller.

## Reversibility

This decision is reversible by deleting `tsdown.config.ts`, restoring `build` to `tsc -p tsconfig.build.json`, restoring TypeScript emit settings in `tsconfig.build.json`, and removing the `tsdown` and `concurrently` dev dependencies and scripts.
