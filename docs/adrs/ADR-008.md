# ADR-008: Include shadcn v4 in base web template

| Status   | Date       |
| -------- | ---------- |
| Accepted | 2026-04-15 |

## Context

The base web template currently includes partial shadcn-ready wiring: Tailwind CSS v4, `components.json`, `@/*` aliases, `cn()`, and a starter `Button`.

shadcn v4 projects need a complete setup across `components.json`, CSS imports and tokens, `radix-ui`, `shadcn`, `lucide-react`, `tw-animate-css`, and starter UI components.

Generated apps should start from a working shadcn baseline instead of requiring manual shadcn setup after scaffolding.

## Decision

Expand the base web template to include a complete shadcn v4 surface.

Keep the scope limited to shadcn wiring, theme CSS, package dependencies, and starter UI components.

## Consequences

Positive:

- Generated web apps can use shadcn immediately after scaffolding.
- Template setup steps are reduced for product UI work.
- Dependencies and config stay aligned with the shadcn v4 setup used by generated apps.

Negative:

- Generated web apps gain more dependencies.
- The base template becomes heavier than a minimal React + Vite + Tailwind app.
- Future shadcn updates may require updating component code and CSS tokens.

## Alternatives Considered

- Minimal shadcn CLI-only config: rejected because it keeps the template only partially ready and still requires manual component setup.
- Keep the existing partial setup: rejected because it advertises shadcn readiness without providing a complete working shadcn v4 baseline.

## Reversibility

Supersede this ADR if the base template later returns to a smaller UI baseline or adopts a different UI component strategy.

Rollback by restoring the previous shadcn-ready template, removing added shadcn dependencies, removing added UI components, and restoring the previous CSS entry and `components.json`.
