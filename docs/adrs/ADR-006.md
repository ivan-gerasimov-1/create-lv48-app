# ADR-006: Avoid project-owned index files

| Status   | Date       |
| -------- | ---------- |
| Accepted | 2026-04-15 |

## Context

The repository currently uses `index.ts` for owned modules under `src/**`.

The base template also uses project-owned `src/index.ts` and `src/index.css` files.

`index` names hide module purpose and encourage barrel-style imports, which makes ownership and dependency direction less explicit.

Some `index` files are framework conventions and should remain. Vite expects a root `index.html`, and Astro uses `src/pages/index.astro` for the `/` route.

## Decision

Use functional filenames for project-owned modules and assets.

Do not create barrel-style `index` files.

Allow framework-required `index` entrypoints only when the convention is external and avoiding it would add needless config.

Document this convention in engineering docs and agent context.

Do not add an automated guard in this ADR version; reviewers must catch regressions.

## Consequences

Positive:

- Imports and files describe their responsibility more clearly.
- Search results show the actual module purpose without opening the file.
- Hidden module surfaces from barrel files are reduced.

Negative:

- Future `index` files can return unless reviewers catch them.
- Some framework-owned `index` names remain, so the convention needs explicit exceptions.

## Alternatives Considered

- Add an automated Vitest or ESLint guard: rejected for this ADR version to keep the change lighter and match the current implementation plan.
- Rename framework entrypoints too: rejected because fighting Vite and Astro conventions would add config and risk for little benefit.
- Keep existing `index` files: rejected because the project wants explicit functional naming.

## Reversibility

Supersede this ADR if the project later chooses an automated guard or decides to rename framework-specific entrypoints.

Rollback by restoring previous filenames, imports, template references, and documentation.
