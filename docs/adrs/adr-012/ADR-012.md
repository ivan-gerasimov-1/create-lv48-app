# ADR-012: Use Clack for interactive CLI prompts

| Status   | Date       | Implementation Plan                                           |
| -------- | ---------- | ------------------------------------------------------------- |
| Proposed | 2026-04-16 | [ADR-012_IMPLEMENTATION-PLAN.md](./ADR-012_IMPLEMENTATION-PLAN.md) |

## Context

The CLI currently uses a project-owned readline prompt wrapper through the prompt controller. The current prompt needs are text input, yes/no confirmation, and a single-choice workspace layout selection.

Keeping these terminal interactions hand-rolled would grow custom behavior around validation, defaults, choice rendering, cancellation, and user-facing prompt output.

Commander is a common choice for argument parsing, help output, flags, positional arguments, subcommands, and non-interactive mode. It does not replace interactive prompt handling, so adding Commander as part of this prompt UI change would not remove the current prompt controller responsibility.

## Decision

Use `@clack/prompts@1.2.0` for interactive CLI prompts.

Keep `createPromptController()` as the public prompt module entrypoint used by the CLI. Preserve the current `TPromptController.collectAnswers(defaultProjectName?)` contract and the `TPromptAnswers` shape returned to generation.

Use Clack prompt primitives for interactive input:

- `text` for project name, target directory, and first app project name.
- `select` for workspace layout.
- `confirm` for install dependencies and initialize git prompts.
- `isCancel` and `cancel` for controlled cancellation.

Install Clack as an exact runtime dependency because the published CLI uses it at runtime.

Do not add Commander, new flags, positional arguments, subcommands, or non-interactive mode as part of this decision. Commander remains a future option when CLI argument parsing becomes a real requirement.

## Consequences

Positive:

- The project owns less terminal input behavior.
- Interactive scaffold UX gets clearer choice and confirmation prompts.
- Cancellation becomes an explicit prompt-layer behavior.
- The existing CLI-to-generation answer contract stays stable.

Negative:

- The published CLI gains a runtime dependency.
- Tests need to mock the prompt wrapper instead of readline behavior.
- Terminal rendering details become library-owned and should not be asserted in tests.

## Alternatives Considered

- Keep readline: lowest dependency count, but custom validation, choice, confirmation, and cancellation behavior would keep growing in project code.
- Use `@inquirer/prompts`: mature and actively maintained, but heavier than needed for this scaffold-style CLI flow.
- Add Commander now: useful for argv parsing and help output, but it does not solve interactive prompts and would add a separate responsibility before the CLI has enough flags or non-interactive behavior to justify it.

## Reversibility

Rollback by restoring the readline-backed prompt implementation, removing `@clack/prompts` from runtime dependencies, and keeping the `TPromptController.collectAnswers(defaultProjectName?)` contract unchanged for the rest of the CLI.
