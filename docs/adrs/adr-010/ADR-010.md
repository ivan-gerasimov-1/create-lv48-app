# ADR-010: Rename template identifier to name

| Status      | Date       | Implementation Plan                                                |
| ----------- | ---------- | ------------------------------------------------------------------ |
| Implemented | 2026-04-16 | [ADR-010_IMPLEMENTATION-PLAN.md](./ADR-010_IMPLEMENTATION-PLAN.md) |

## Context

[ADR-009](../adr-009/ADR-009.md) introduced typed template definitions as the source of truth for local template metadata.

The current metadata uses `id: "base"` as the template identifier. That same value is exposed through preset metadata, used by the preset registry, appears in CLI output, and drives the derived `filesRoot` path.

The term `id` is now doing both public metadata work and internal lookup work. `name` is a clearer authoring contract for local templates, especially while the registry has a single built-in `base` template and no remote identifier namespace.

## Decision

Use `name` as the canonical template and preset identifier.

Template metadata MUST expose `name`, not `id`. Preset metadata MUST mirror that name. Preset registry APIs MUST use name terminology, including `defaultPresetName` and `getPresetByName`.

`defineTemplate(config)` MUST derive the internal `filesRoot` from `config.name`. The base template remains named `base`, and generated assets remain under `templates/base/files`.

Do not keep a compatibility `id` alias.

## Consequences

Positive:

- Template authoring language becomes clearer and less overloaded.
- Template metadata, preset metadata, registry lookup, CLI output, and docs use the same identifier term.
- TypeScript catches remaining callers that still expect `id`.

Negative:

- This is a breaking TypeScript contract for any consumer that imports template or preset metadata and reads `id`.
- Any downstream code using `defaultPresetId` or `getPresetById` must rename to the new name-based API.

## Alternatives Considered

- Rename only template metadata: rejected because preset registry and CLI output would keep `id` terminology and preserve the mismatch.
- Keep `id` as a deprecated compatibility alias: rejected because the current API surface is small and the alias would add drift without a clear compatibility need.
- Keep `id` everywhere: rejected because it leaves the overloaded identifier language in place after the naming concern has been identified.

## Reversibility

Rollback by restoring `id` on template and preset metadata, restoring `defaultPresetId` and `getPresetById`, changing `defineTemplate()` back to `config.id`, and reverting documentation that describes the name-based contract.
