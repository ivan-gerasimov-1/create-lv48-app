# ADR-003: Support single and multi-project workspace layouts

## Status

Implemented

## Implementation

See [ADR-003_IMPLEMENTATION-PLAN.md](./ADR-003_IMPLEMENTATION-PLAN.md).

## Context

The scaffold currently generates one app set directly under `apps/`:

```txt
apps/
  web/
  site/
  api/
packages/
```

This works for a single product project, but it does not support a monorepo that contains multiple app projects with the same workspace names, for example:

```txt
apps/
  project-1/
    web/
    site/
    api/
  project-2/
    web/
    site/
    api/
packages/
```

The current root npm workspace glob is `apps/*`, which only matches direct app workspaces. A nested multi-project layout requires `apps/*/*`.

The current package JSON transform derives app package names from the second path segment under `apps/`. If app package manifests move to `apps/project-1/web/package.json`, that approach would name the package from `project-1` instead of the actual app workspace. Multi-project layout therefore needs layout-aware package naming.

There are two different naming concerns:

- root package scope for shared packages under `packages/`
- app project scope for app workspaces under `apps/<project>/`

Keeping these scopes separate avoids collisions between app projects while preserving a stable namespace for shared packages.

## Decision

Support two workspace layouts in the base preset:

- `single`, the default and current output shape
- `multi`, an optional nested app-project output shape

Keep single-project output backward-compatible:

```txt
apps/
  web/
  site/
  api/
packages/
```

Single-project root workspaces remain:

```json
["apps/*", "packages/*"]
```

Single-project app package names continue to use the root package scope:

```txt
@<rootPackage>/web
@<rootPackage>/site
@<rootPackage>/api
```

In multi-project layout, generate the first app project under `apps/<appProjectName>/`:

```txt
apps/
  <appProjectName>/
    web/
    site/
    api/
packages/
```

Multi-project root workspaces use:

```json
["apps/*/*", "packages/*"]
```

Multi-project app package names use the app project scope:

```txt
@<appProjectName>/web
@<appProjectName>/site
@<appProjectName>/api
```

Shared packages always use the root package scope:

```txt
packages/eslint-config -> @<rootPackage>/eslint-config
```

Add a CLI layout prompt with `single` as the default. Ask for `First app project name` only when the user selects `multi`, defaulting to `project-1`.

Implement layout support by making destination path mapping and package JSON transforms aware of generation context.

## Consequences

Positive:

- Existing default scaffold remains stable for current users.
- Multi-project repositories can contain repeated app workspace names without package name collisions.
- Shared packages keep one root-level namespace independent from app projects.
- The base preset can cover both layouts without introducing preset selection before it is needed.

Negative:

- Prompt answers, path mapping, placeholders, and package JSON transforms become layout-aware.
- App package naming differs between single and multi layouts.
- Generated docs and tests must cover both layouts to avoid drift.

## Alternatives Considered

- Always switch to multi-project layout: simpler long-term structure, but it is a breaking change for current scaffold users.
- Add a separate preset for multi-project layout: cleaner conceptual separation, but the CLI does not currently expose preset selection and the template difference is mostly path and manifest transformation.
- Keep multi-project app packages under the root scope, such as `@root/web`: backward-looking naming, but it collides as soon as a second app project also has `web`.
- Use nested package names such as `@root/project-1/web`: visually expressive, but npm package names cannot contain two slash-separated segments after the scope.

## Reversibility

This decision is reversible by removing the layout prompt and the multi-project branches in path mapping, package JSON transforms, placeholders, generated docs, and tests. Single-project behavior remains the rollback baseline.
