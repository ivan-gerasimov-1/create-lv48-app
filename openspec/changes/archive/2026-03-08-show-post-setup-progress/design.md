## Context

Сейчас `runCli` завершает scaffold и затем вызывает `postSetupExecutor.run(...)` без промежуточных пользовательских сообщений. Внутри post-setup шагов stdout чаще всего молчит до первого вывода дочернего процесса, а некоторые команды или среды могут не выводить ничего заметного довольно долго.

## Goals / Non-Goals

**Goals:**
- Сделать длительные optional post-setup шаги явно видимыми для пользователя.
- Сохранить текущую последовательность выполнения и обработку ошибок.
- Избежать новых зависимостей и сложного интерактивного спиннера.

**Non-Goals:**
- Перестраивать prompt flow или summary flow.
- Добавлять real-time процентный прогресс для `npm install`.
- Менять набор post-setup шагов или их порядок.

## Decisions

- Использовать текстовые progress-сообщения через уже существующий logger вместо нового spinner/UI слоя.
  Альтернатива: терминальный спиннер. Отклонена, потому что это добавляет зависимость, усложняет тесты и хуже ведет себя в non-TTY/CI средах.
- Инжектировать `onActionStart` callback в post-setup executor.
  Альтернатива: логировать только вокруг `postSetupExecutor.run(...)` в `runCli`. Отклонена, потому что executor лучше знает, какие шаги реально выбраны и в каком порядке выполняются.
- Логировать только старт выбранных шагов; завершение и сбои уже отражаются в финальном summary.
  Альтернатива: отдельные start/end логи для каждого шага. Отклонена как более шумная для текущего CLI.

## Risks / Trade-offs

- [Повторение информации] -> Финальный summary уже содержит статусы шагов; mitigation: progress-сообщения ограничены только стартом долгих операций.
- [Слишком общий текст] -> Пользователь может не понять, сколько ждать; mitigation: сообщения называют конкретное действие и цель (`npm install`, `git init`).
- [Будущие шаги забудут про progress] -> Mitigation: callback проходит через общий helper `runOptionalAction`, а не добавляется точечно в одном месте.

## Migration Plan

Изменение additive и обратимое: обновить CLI-код и тесты, затем синхронизировать delta spec в основной spec. Rollback — удалить progress callback и соответствующие spec/test изменения без миграций данных.
